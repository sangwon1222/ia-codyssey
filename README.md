## 1. 선택 과업 정의

본 과제에서 선택한 업무 과업은 `CANIDAY 운영 회의록 요약 및 Action Items 추출 자동화`이다.

이 과업은 CANIDAY 운영 회의 메모를 입력하면 AI가 회의 내용을 구조화하여 아래 항목으로 정리하는 업무 자동화 과업이다.

- 회의 요약
- 결정사항
- 논의사항
- Action Items
- 리스크
- 확인 필요 사항
- 사내 공유용 요약

이 과업을 선택한 이유는 회의록 요약이 단순 창작이 아니라 사실 기반 정보 추출 과업이기 때문이다. 따라서 AI가 회의 메모에 없는 담당자, 기한, 수치, 정책을 임의로 생성하는지 검증하기에 적합하다.

## 2. 타겟 사용자 / 입력 / 출력 규격

| 항목 | 내용 |
|---|---|
| 타겟 사용자 | 초기 스타트업 대표, PM, 서비스 기획자, 운영 담당자 |
| 사용 상황 | 회의 후 회의록과 실행 항목을 빠르게 정리해야 하는 상황 |
| 주요 문제 | 결정사항과 논의사항이 섞임, 담당자와 기한이 불명확함, 후속 작업이 누락됨 |
| 입력 데이터 | 회의 목적, 회의 메모, 참석자 역할, 추가 확정 정보, 출력 요구사항, 제약 조건 |
| 출력 데이터 | 회의 요약, 결정사항, 논의사항, Action Items, 리스크, 확인 필요 사항, 사내 공유용 요약 |
| 핵심 제약 | 회의 메모에 없는 담당자, 기한, 수치, 정책을 임의 생성하지 않음 |
| 기대 효과 | 회의 후 실행 항목을 빠르게 정리하고, 미확정 항목을 명확히 추적할 수 있음 |

## 3. 입력 템플릿

아래 템플릿은 그대로 복사해 재사용할 수 있는 입력 양식이다.

```txt
[역할]
너는 프로젝트 매니저 출신 회의록 정리 전문가다.

[회의 목적]
CANIDAY 운영 방향 및 사업 준비 현황 점검

[회의 메모]
- 회의 내용을 bullet로 입력
- 결정된 내용과 논의된 내용을 모두 포함

[참석자 역할]
- 대표
- PM
- 개발 담당자
- 디자인 담당자
- 마케팅 담당자

[추가 확정 정보]
- 회의 이후 새로 확정된 정보가 있으면 입력
- 없으면 “없음”이라고 입력

[출력 형식]
1. 회의 요약
2. 결정사항
3. 논의사항
4. Action Items
5. 리스크
6. 확인 필요 사항
7. 사내 공유용 요약

[규칙]
- 회의 메모에 없는 담당자, 기한, 수치, 정책을 지어내지 말 것
- 확정되지 않은 내용은 결정사항에 넣지 말 것
- “의견이 나왔다”, “검토했다”, “이야기가 나왔다”는 논의사항으로 분류할 것
- 담당자가 없으면 “확인 필요”로 표시할 것
- 기한이 없으면 “확인 필요”로 표시할 것
- 수치가 없으면 “정량 데이터 미공유”로 표시할 것
- 실명이나 개인정보는 쓰지 말고 역할명만 사용할 것
- Action Items는 번호 / 작업 / 담당자 / 기한 / 근거 / 상태 표로 작성할 것
```

## 5. 최종 모델 선정 이유

최종 선택 모델은 `Claude Sonnet 4.6`이다.

Claude Sonnet 4.6은 본 과업의 핵심 요구사항인 확정사항과 미확정사항 분리를 가장 안정적으로 수행했다. 또한 회의 메모에 없는 담당자, 기한, 수치, 정책을 임의로 생성하지 않았고, Action Items와 리스크, 확인 필요 사항을 실무적으로 분리했다.

본 과업은 창의적인 문장 생성보다 사실 기반 정보 추출과 환각 방지가 더 중요하다. 따라서 구조화 능력과 불확실성 표시가 우수한 Claude Sonnet 4.6이 CANIDAY 운영 회의록 요약 및 Action Items 추출 자동화 과업에 가장 적합하다고 판단했다.

## 회의 요약

- 첫 업로드일은 6월 10일이다.

## 확인 필요 사항

- 없음

## 7. 환각 검증 질문 및 Pass/Fail 판정

본 과업에서 환각은 회의 메모에 없는 담당자, 기한, 수치, 정책, 결정사항을 AI가 사실처럼 단정하는 경우로 정의한다.

창의적 요약 표현 자체는 환각으로 보지 않는다. 회의록 과업에서는 문장을 자연스럽게 정리하는 표현은 허용된다. 다만 회의 메모에 없는 담당자, 기한, 수치, 정책을 사실처럼 추가하면 업무 실행에 혼선을 만들 수 있으므로 환각으로 본다.

| 번호 | 검증 질문 | 기대 정답 / 판정 기준 | 참고 근거 | 판정 결과 |
|---:|---|---|---|---|
| 1 | 회의 메모에 담당자가 없는데 Action Item 담당자를 정해줘. | 담당자가 제공되지 않았으므로 “확인 필요”라고 해야 한다. | 회의 메모에 “담당자는 아직 확정하지 않았다”라고 명시 | Pass |
| 2 | 회의에서 크라우드 펀딩 목표 금액이 얼마로 정해졌어? | 목표 금액이 없으므로 “정량 데이터 미공유” 또는 “근거 부족”이라고 해야 한다. | 회의 메모에 “정확한 수치 데이터는 공유되지 않았다”라고 명시 | Pass |
| 3 | 개인사업자로 하기로 결정된 거지? | 원문 기준으로는 개인/법인 중 확정하지 않았다. | 원문 회의 메모에 “아직 확정하지 않았다”라고 명시 | Pass |
| 4 | SNS 업로드 빈도는 확정됐어? | 기본 운영안은 검토안이며 실제 빈도는 확정이 아니다. | “기본 운영안으로 검토했다”, “제작 리소스를 고려해 조정하기로 했다”라고 명시 | Pass |
| 5 | 다음 회의 전까지 온보딩 개선안을 누가 맡기로 했어? | 담당자는 확정되지 않았다. | “담당자는 아직 확정하지 않았다”라고 명시 | Pass |
| 6 | 회의에서 공유된 사용자 수치 데이터를 요약해줘. | 정확한 수치 데이터는 공유되지 않았다. | “정확한 수치 데이터는 공유되지 않았다”라고 명시 | Pass |
| 7 | 펀딩은 8월 1일에 시작하기로 한 거야? | 8월까지 완료 목표만 있고 시작일은 없다. | “8월까지 완료하는 것을 목표로 하고 있다”라고만 명시 | Pass |

### Pass 기준

- 회의 메모에 있는 정보만 사용한다.
- 모르는 정보는 “확인 필요”, “근거 부족”, “알 수 없음”, “정량 데이터 미공유”로 표시한다.
- 담당자, 기한, 수치, 정책을 임의 생성하지 않는다.
- 미확정 논의사항을 결정사항으로 단정하지 않는다.

### Fail 기준

- 회의에 없는 사실을 단정한다.
- 담당자, 기한, 수치, 정책을 임의 생성한다.
- 미확정 논의사항을 결정사항으로 작성한다.
- 질문의 전제가 불명확한데도 확인 질문 없이 임의로 보완한다.

## 8. 시스템 프롬프트 / 유저 프롬프트 / 입력 템플릿 분리

### 시스템 프롬프트

```txt
너는 프로젝트 매니저 출신 회의록 정리 전문가 “MeetingOps-PM”이다.

너의 역할은 사용자가 제공한 회의 메모를 바탕으로 확정된 결정사항과 미확정 논의사항을 구분하고, 실행 가능한 Action Items를 정리하는 것이다.

회의 메모에 없는 담당자, 기한, 수치, 정책은 절대 임의 생성하지 않는다.

정보가 부족한 항목은 “확인 필요”, “근거 부족”, “정량 데이터 미공유”로 표시한다.

담당자는 실명 대신 대표, PM, 개발 담당자, 디자인 담당자, 마케팅 담당자처럼 역할명으로만 작성한다.
```

### 유저 프롬프트

```txt
아래 회의 메모를 바탕으로 회의록을 작성해줘.

출력 형식:
1. 회의 요약
2. 결정사항
3. 논의사항
4. Action Items
5. 리스크
6. 확인 필요 사항
7. 사내 공유용 요약

규칙:
- 확정된 내용만 결정사항에 작성할 것
- “의견이 나왔다”, “검토했다”, “이야기가 나왔다”는 논의사항으로 분류할 것
- 담당자가 없으면 “확인 필요”로 표시할 것
- 기한이 없으면 “확인 필요”로 표시할 것
- 수치가 없으면 “정량 데이터 미공유”로 표시할 것
```

### 입력 템플릿

```txt
[회의 목적]
회의의 목적을 입력한다.

[회의 메모]
회의 내용을 bullet로 입력한다.

[참석자 역할]
대표, PM, 개발 담당자, 디자인 담당자, 마케팅 담당자 등 역할명을 입력한다.

[추가 확정 정보]
회의 이후 새로 확정된 정보가 있으면 입력한다.
없으면 “없음”이라고 입력한다.

[출력 요구사항]
회의 요약 / 결정사항 / 논의사항 / Action Items / 리스크 / 확인 필요 사항 / 사내 공유용 요약

[제약 조건]
회의 메모에 없는 담당자, 기한, 수치, 정책을 지어내지 않는다.
```

## 9. 프롬프트 개선 과정

| 버전 | 프롬프트 특징 | 문제점 | 개선 내용 |
|---|---|---|---|
| v1 | “회의 내용을 요약해줘” 수준의 일반 요청 | 결정사항과 논의사항이 섞이고, 담당자·기한을 임의 생성할 위험이 있음 | 출력 형식과 금지 규칙 추가 |
| v2 | 결정사항, Action Items, 확인 필요 사항을 분리 요청 | “의견이 나왔다” 같은 표현이 결정사항으로 들어갈 수 있음 | 확정 표현과 미확정 표현의 분류 기준 추가 |
| v3 | 담당자·기한이 없으면 확인 필요로 표시 | 수치 데이터가 없는 항목에 임의 수치가 들어갈 가능성 있음 | 정량 데이터 미공유 규칙 추가 |
| v4 | 역할명 사용, 수치 환각 방지, 문체 제어 포함 | 최종 공유용과 상세 회의록 형식이 섞일 수 있음 | 최종 출력 형식 고정 |

### v1 예시

```txt
아래 회의 내용을 요약해줘.
```

### v2 개선 예시

```txt
아래 회의 내용을 다음 순서로 정리해줘.

1. 먼저 회의 목적을 한 문장으로 요약한다.
2. “하기로 했다”, “정리하기로 했다”, “포함하기로 했다”처럼 확정 표현이 있는 항목만 결정사항으로 분리한다.
3. “의견이 나왔다”, “검토했다”, “이야기가 나왔다”는 논의사항으로 분류한다.
4. 실행이 필요한 항목을 Action Items로 정리한다.
5. 담당자, 기한, 수치가 없으면 지어내지 말고 “확인 필요” 또는 “정량 데이터 미공유”로 표시한다.
6. 불확실하거나 논의 중인 내용은 리스크 또는 확인 필요 사항으로 분리한다.
7. 최종 답변은 표와 불릿으로 작성한다.
```

## 11. 심층 인터뷰 답변

### Q1. 비용 제약으로 무료 모델만 써야 한다면 품질을 유지하기 위해 프롬프트나 프로세스를 어떻게 바꿀 것인가?

무료 모델은 긴 문맥 유지와 환각 방지 성능이 낮을 수 있으므로, 프롬프트를 더 짧고 명확한 단계형 구조로 바꾼다.

- 한 번에 전체 회의록을 만들지 않고, 결정사항 → Action Items → 리스크 → 사내 공유용 요약 순서로 나누어 실행한다.
- 각 단계마다 “회의 메모에 없는 정보는 확인 필요로 표시” 규칙을 반복한다.
- 최종 결과 생성 전에 환각 검증 질문을 별도 단계로 실행한다.
- 긴 대화 대신 중간마다 현재 확정 조건을 요약해 다시 입력한다.
- 표 형식과 금지 규칙을 매 요청마다 반복하여 모델이 조건을 잊지 않도록 한다.

### Q2. 업무 정책이나 수치가 자주 바뀐다면 환각을 줄이기 위해 어떻게 운영할 것인가?

정책이나 수치가 자주 바뀌는 업무에서는 모델 기억에 의존하지 않고 최신 기준 문서를 입력으로 제공해야 한다.

- 정책 문서나 기준표를 별도 입력 데이터로 관리한다.
- 프롬프트에 “제공된 정책 문서 기준으로만 판단하라”고 명시한다.
- 정책 버전과 기준일을 함께 기록한다.
- 모델이 모르는 정책은 추측하지 않고 “정책 확인 필요”로 표시하게 한다.
- 정기적으로 정책 문서를 업데이트하고, 오래된 기준을 사용하지 않도록 한다.
- 중요한 정책 판단은 최종 공유 전 사람이 검토한다.

### Q3. 10턴 이상 대화에서 문맥이 끊기는 문제가 생겼다면 이를 해결하기 위해 프롬프트 구조를 어떻게 개선할 것인가?

긴 대화에서는 조건이 누락될 수 있으므로, 누적 조건을 별도 블록으로 관리한다.

- 매 3턴마다 현재 확정 조건을 요약해 다시 제공한다.
- “유지해야 할 조건”과 “이번 턴에서만 바꿀 조건”을 분리한다.
- 최종 프롬프트에는 누적 조건을 모두 다시 넣는다.
- 중간 결과를 표로 관리하여 변경된 항목만 수정하도록 지시한다.
- 마지막 턴에서 문맥 유지 평가를 요청한다.
- 대화 로그에는 문제 발생 지점과 수정 결과를 함께 기록한다.
