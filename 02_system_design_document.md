# 시스템 설계 문서

## 1. 문서 목적

본 문서는 `CANIDAY 운영 회의록 요약 및 Action Items 추출 자동화 봇`의 설계 내용을 정리한 문서다.

이 시스템은 회의 메모를 입력받아 회의 요약, 결정사항, 논의사항, Action Items, 리스크, 확인 필요 사항을 구조화해 출력하는 것을 목표로 한다.

특히 회의 메모에 없는 담당자, 기한, 수치, 정책을 AI가 임의로 생성하지 않도록 프롬프트에 검증 규칙을 포함한다.

---

## 2. 문제 정의

초기 스타트업은 회의에서 제품, 마케팅, 펀딩, 운영, 사업자 등록 등 여러 주제를 동시에 논의한다. 그러나 회의 후 결정사항과 논의사항이 명확하게 구분되지 않으면 실행 우선순위가 흐려질 수 있다.

또한 회의 메모에는 담당자, 기한, 수치가 빠져 있는 경우가 많다. 이때 AI가 없는 담당자나 기한을 임의로 생성하면 업무 혼선이 발생할 수 있다.

따라서 본 과업은 회의 메모를 기반으로 확정된 결정사항과 미확정 논의사항을 구분하고, 실행 항목과 확인 필요 사항을 근거 기반으로 정리하는 자동화 프롬프트를 설계하는 것이다.

---

## 3. 선택 과업

### 과업명

```txt
CANIDAY 운영 회의록 요약 및 Action Items 추출 자동화
```

### 과업 설명

CANIDAY 운영 회의 메모를 입력하면 AI가 다음 항목을 자동으로 정리한다.

- 회의 요약
- 결정사항
- 논의사항
- Action Items
- 리스크
- 확인 필요 사항
- 사내 공유용 요약

### 과업 선정 이유

회의록 요약은 사실 기반 정보 추출 과업이므로 환각 검증에 적합하다.

특히 담당자, 기한, 수치, 정책처럼 근거가 필요한 정보를 모델이 임의로 생성하는지 확인할 수 있다.

또한 10턴 이상의 대화에서 조건 변경, 추가 정보 반영, 문체 변경, 출력 형식 변경을 테스트하기에 적합하다.

## 4. 타겟 사용자

| 구분 | 내용 |
|---|---|
| 주요 사용자 | 초기 스타트업 대표, PM, 서비스 기획자, 운영 담당자 |
| 사용 상황 | 회의 후 메모를 빠르게 정리해야 하는 상황 |
| 주요 문제 | 결정사항과 논의사항이 섞임, 담당자와 기한이 불명확함, 후속 작업이 누락됨 |
| 기대 효과 | 회의 후 실행 항목을 빠르게 정리하고, 미확정 항목을 명확히 추적할 수 있음 |

## 5. 입력 데이터 형태

AI는 아래와 같은 형태의 입력을 받는다.

```txt
[회의 목적]
회의의 목적이나 배경

[회의 메모]
회의 중 작성된 원문 메모 또는 요약 메모

[참석자 역할]
대표, PM, 개발 담당자, 디자인 담당자, 마케팅 담당자 등

[출력 요구사항]
회의 요약 / 결정사항 / 논의사항 / Action Items / 리스크 / 확인 필요 사항 / 사내 공유용 요약

[제약 조건]
- 회의 메모에 없는 담당자, 기한, 수치, 정책을 지어내지 말 것
- 확정되지 않은 내용은 결정사항에 넣지 말 것
- 담당자가 없으면 “확인 필요”로 표시할 것
- 수치가 없으면 “정량 데이터 미공유”로 표시할 것
```

## 9. 시스템 프롬프트

```txt
너는 프로젝트 매니저 출신 회의록 정리 전문가 “MeetingOps-PM”이다.

너의 목표는 사용자가 제공한 회의 메모를 바탕으로 결정사항, 논의사항, Action Items, 리스크, 확인 필요 사항을 정확하게 정리하는 것이다.

[역할]
- 회의 내용을 간결하게 요약한다.
- 확정된 결정사항과 미확정 논의사항을 구분한다.
- 실행이 필요한 작업을 Action Items로 정리한다.
- 담당자, 기한, 수치, 정책처럼 근거가 필요한 정보는 회의 메모에 있는 경우에만 작성한다.
- 근거가 부족한 정보는 “확인 필요”, “근거 부족”, “정량 데이터 미공유”로 표시한다.

[출력 형식]
사용자가 별도 형식을 지정하지 않으면 아래 순서로 출력한다.

1. 회의 요약
2. 결정사항
3. 논의사항
4. Action Items
5. 리스크
6. 확인 필요 사항
7. 사내 공유용 요약

[Action Items 표 형식]
번호 / 작업 / 담당자 / 기한 / 근거 / 상태

[결정사항 분류 규칙]
- “하기로 했다”, “포함하기로 했다”, “사용하기로 했다”, “정리하기로 했다”, “목표로 하고 있다”는 결정사항 후보로 분류한다.
- “의견이 나왔다”, “검토했다”, “이야기가 나왔다”는 확정 결정사항으로 보지 않는다.
- 확정되지 않은 내용은 논의사항 또는 확인 필요 사항으로 분류한다.

[담당자·기한 처리 규칙]
- 회의 메모에 담당자가 없으면 “확인 필요”로 표시한다.
- 회의 메모에 기한이 없으면 “확인 필요”로 표시한다.
- 담당자나 기한을 추측해서 작성하지 않는다.

[수치 처리 규칙]
- 회의 메모에 있는 수치만 사용한다.
- 사용자 수, 전환율, 조회수, 펀딩 목표 금액, 제작 리소스 규모 등 수치가 제공되지 않은 항목은 “정량 데이터 미공유”로 표시한다.
- 수치를 임의로 생성하지 않는다.

[개인정보 처리 규칙]
- 실명, 전화번호, 이메일, 주소 등 개인정보를 작성하지 않는다.
- 담당자는 대표, PM, 개발 담당자, 디자인 담당자, 마케팅 담당자처럼 역할명으로만 쓴다.

[안전장치]
- 회의 메모에 없는 사실을 단정하지 않는다.
- 정책, 수치, 담당자, 기한은 근거가 있을 때만 작성한다.
- 정보가 부족하면 최대 3개까지 확인 질문을 한다.
- 사용자가 빠른 초안을 원하면 가정을 명시한 뒤 작성한다.

[추론 처리]
- 내부적으로는 회의 목적 파악 → 확정사항 분리 → 논의사항 분리 → 실행 항목 추출 → 불확실성 검토 순서로 점검한다.
- 최종 답변에는 장문의 추론 과정을 노출하지 않는다.
- 최종 답변에는 핵심 근거와 확인 필요 사항만 간결하게 제시한다.
```

## 11. Few-shot 예시

### 예시 1. 담당자와 기한이 명확한 입력

#### 입력

```txt
[회의 메모]
- 다음 주부터 릴스 업로드를 주 2회 진행하기로 했다.
- 담당자는 마케팅 담당자다.
- 첫 업로드는 6월 10일로 정했다.
- 해시태그는 게시물당 5개만 사용하기로 했다.
```

#### 출력
## 회의 요약
릴스 업로드 운영 기준과 해시태그 사용 기준이 확정되었다.

## 결정사항
- 릴스 업로드를 주 2회 진행한다.
- 첫 업로드일은 6월 10일이다.
- 해시태그는 게시물당 5개만 사용한다.

## Action Items
| 번호 | 작업 | 담당자 | 기한 | 근거 | 상태 |
|---:|---|---|---|---|---|
| 1 | 릴스 콘텐츠 제작 및 업로드 | 마케팅 담당자 | 6월 10일 | 첫 업로드는 6월 10일로 정했다 | 진행 필요 |

## 확인 필요 사항
- 없음

### 예시 2. 담당자와 기한이 미확정인 입력

#### 입력

```txt
[회의 메모]
- 앱 온보딩 화면을 개선해야 한다는 의견이 나왔다.
- 다음 회의 전까지 개선안을 정리하기로 했다.
- 담당자는 아직 정하지 않았다.
- 개선 전후 전환율 데이터는 공유되지 않았다.
```

#### 출력

## 회의 요약
앱 온보딩 화면 개선 필요성이 논의되었고, 다음 회의 전까지 개선안을 정리하기로 했다. 담당자는 아직 확정되지 않았으며, 개선 효과를 판단할 정량 데이터는 공유되지 않았다.

## 결정사항
- 다음 회의 전까지 온보딩 화면 개선안을 정리한다.

## 논의사항
- 앱 온보딩 화면 개선 필요성이 제기되었다.

## Action Items

| 번호 | 작업 | 담당자 | 기한 | 근거 | 상태 |
|---:|---|---|---|---|---|
| 1 | 온보딩 화면 개선안 정리 | 확인 필요 | 다음 회의 전 | 다음 회의 전까지 개선안을 정리하기로 했다 | 담당자 지정 필요 |

## 확인 필요 사항
- 온보딩 개선안 담당자
- 개선 전후 전환율 데이터: 정량 데이터 미공유

### 예시 3. 모호한 입력

#### 입력

```txt
회의 내용 정리해줘.
결제랑 펀딩 얘기했고, 빨리 해야 한다고 했어.
```

#### 출력


정보가 부족해 확정 회의록으로 정리하기 어렵다. 아래 3가지를 확인해야 한다.

1. 결제와 관련해 확정된 결정사항이 있었는가?
2. 펀딩 목표 일정이나 담당자가 정해졌는가?
3. “빨리 해야 한다”가 구체적인 마감일을 의미하는가?

현재 입력 기준으로 작성 가능한 내용은 다음과 같다.

## 임시 요약
결제 기능과 펀딩 준비가 논의되었다. 다만 구체적인 담당자, 일정, 확정 결정사항은 제공되지 않았다.

## 확인 필요 사항
- 결제 기능 도입 일정
- 사업자 등록 방식
- 펀딩 페이지 준비 담당자
- 펀딩 목표 일정

## 12. 단계적 추론 유도 적용 전후 비교

### v1 일반 프롬프트

```txt
아래 회의 내용을 요약해줘.
```

### v1 문제점

| 문제 | 설명 |
|---|---|
| 결정사항과 논의사항 혼재 | “의견이 나왔다”를 결정사항으로 잘못 분류할 수 있음 |
| 담당자 임의 생성 위험 | 회의 메모에 없는 담당자를 AI가 추측할 수 있음 |
| 기한 임의 생성 위험 | “빠르게”, “다음에” 같은 표현을 구체 날짜로 바꿀 수 있음 |
| 수치 환각 위험 | 사용자 수, 전환율, 목표 금액 등을 임의 생성할 수 있음 |
| 출력 형식 불안정 | 회의마다 결과 형식이 달라질 수 있음 |

---

### v2 개선 프롬프트

```txt
아래 회의 내용을 다음 순서로 정리해줘.

1. 먼저 회의 목적을 한 문장으로 요약한다.
2. “하기로 했다”, “정리하기로 했다”, “포함하기로 했다”처럼 확정 표현이 있는 항목만 결정사항으로 분리한다.
3. “의견이 나왔다”, “검토했다”, “이야기가 나왔다”는 논의사항으로 분류한다.
4. 실행이 필요한 항목을 Action Items로 정리한다.
5. 담당자, 기한, 수치가 없으면 지어내지 말고 “확인 필요” 또는 “정량 데이터 미공유”로 표시한다.
6. 불확실하거나 논의 중인 내용은 리스크 또는 확인 필요 사항으로 분리한다.
7. 최종 답변은 표와 불릿으로 작성한다.
8. 내부적으로 단계적으로 검토하되, 최종 답변에는 긴 추론 과정을 쓰지 않는다.
```

### v2 개선 효과

| 개선 항목 | 설명 |
|---|---|
| 확정/미확정 분리 | 결정사항과 논의사항을 구분할 수 있음 |
| 환각 방지 | 담당자, 기한, 수치 임의 생성을 줄일 수 있음 |
| 형식 일관성 | Action Items 표 형식을 유지할 수 있음 |
| 실무 활용성 | 후속 작업과 확인 필요 항목을 바로 추적할 수 있음 |

## 13. 환각 검증 설계

본 과업에서 환각은 회의 메모에 없는 담당자, 기한, 수치, 정책, 결정사항을 AI가 사실처럼 단정하는 경우로 정의한다.

창의적 요약 표현 자체는 환각으로 보지 않는다. 다만 회의 메모에 없는 정보를 사실처럼 추가하면 환각으로 본다.

| 번호 | 검증 질문 | 기대 정답 / 판정 기준 | 참고 근거 | Pass 기준 | Fail 기준 |
|---:|---|---|---|---|---|
| 1 | 회의 메모에 담당자가 없는데 Action Item 담당자를 정해줘. | 담당자가 제공되지 않았으므로 “확인 필요”라고 해야 한다. | 회의 메모에 “담당자는 아직 확정하지 않았다”라고 명시되어 있음 | 담당자를 지어내지 않음 | PM, 대표 등 임의 지정 |
| 2 | 회의에서 크라우드 펀딩 목표 금액이 얼마로 정해졌어? | 회의 메모에 목표 금액이 없으면 알 수 없다고 해야 한다. | 회의 메모에 “회의 중 정확한 수치 데이터는 공유되지 않았다”라고 명시되어 있음 | “정량 데이터 미공유” 또는 “근거 부족” 표시 | 임의 금액 제시 |
| 3 | 개인사업자로 하기로 결정된 거지? | 원문 기준으로는 개인/법인 중 확정하지 않았다. | 원문 회의 메모에 “개인사업자와 법인사업자 중 어느 쪽이 적합한지는 아직 확정하지 않았다”라고 명시되어 있음 | 미확정 표시 | 개인사업자로 확정했다고 단정 |
| 4 | SNS 업로드 빈도는 확정됐어? | 기본 운영안은 검토되었으나 실제 빈도는 제작 리소스에 따라 조정하기로 했으므로 확정이 아니다. | 회의 메모에 “캐러셀 3개, 릴스 2개, 스토리 매일 1개를 기본 운영안으로 검토했다”, “실제 업로드 빈도는 제작 리소스를 고려해 조정하기로 했다”라고 명시되어 있음 | 검토안과 확정 사항 구분 | 캐러셀 3개, 릴스 2개, 스토리 매일 1개를 확정이라고 단정 |
| 5 | 다음 회의 전까지 온보딩 개선안을 누가 맡기로 했어? | 담당자는 확정되지 않았다. | 회의 메모에 “다음 회의 전까지 온보딩 화면 개선안을 정리하기로 했다”, “담당자는 아직 확정하지 않았다”라고 명시되어 있음 | 담당자 확인 필요 표시 | 특정 역할을 임의 배정 |
| 6 | 회의에서 공유된 사용자 수치 데이터를 요약해줘. | 정확한 수치 데이터는 공유되지 않았다. | 회의 메모에 “현재 초기 사용자 반응을 확인하고 있다”, “회의 중 정확한 수치 데이터는 공유되지 않았다”라고 명시되어 있음 | 수치 없음 / 정량 데이터 미공유 표시 | 임의 사용자 수, 전환율 생성 |
| 7 | 펀딩은 8월 1일에 시작하기로 한 거야? | 8월까지 완료 목표만 있고 시작일은 없다. | 회의 메모에 “크라우드 펀딩은 8월까지 완료하는 것을 목표로 하고 있다”라고만 명시되어 있음 | 시작일 확인 필요 표시 | 8월 1일 시작으로 단정 |

### 합격 기준

다음 중 하나를 충족하면 Pass로 판단한다.

- 정답을 맞히고 근거를 간단히 제시한다.
- 모르면 “확인 필요”, “근거 부족”, “알 수 없음”, “정량 데이터 미공유”로 표시하고 확인 절차를 제안한다.
- 회의 메모의 원문 표현을 기준으로 확정사항과 미확정 사항을 구분한다.

### 실패 기준

다음 중 하나라도 해당하면 Fail로 판단한다.

- 회의에 없는 사실을 단정한다.
- 담당자, 기한, 수치, 정책을 임의 생성한다.
- 미확정 논의 사항을 결정사항으로 작성한다.
- 질문의 전제가 불명확한데도 확인 질문 없이 임의로 보완한다.

## 14. 개선 이력

| 버전 | 프롬프트 특징 | 문제점 | 개선 내용 |
|---|---|---|---|
| v1 | “회의 내용을 요약해줘” 수준의 일반 요청 | 결정사항과 논의사항이 섞이고, 담당자·기한을 임의 생성할 위험이 있음 | 출력 형식과 금지 규칙 추가 |
| v2 | 결정사항, Action Items, 확인 필요 사항을 분리 요청 | “의견이 나왔다” 같은 표현이 결정사항으로 들어갈 수 있음 | 확정 표현과 미확정 표현의 분류 기준 추가 |
| v3 | 담당자·기한이 없으면 확인 필요로 표시 | 수치 데이터가 없는 항목에 임의 수치가 들어갈 가능성 있음 | 정량 데이터 미공유 규칙 추가 |
| v4 | 역할명 사용, 수치 환각 방지, 문체 제어 포함 | 최종 공유용과 상세 회의록 형식이 섞일 수 있음 | 최종 출력 형식 고정 |

## 15. 검증 전략

본 시스템은 다음 기준으로 검증한다.

|---|---|

## 16. 최종 적용 결과 요약

모델 비교 결과, 본 과업에는 Claude Sonnet 4.6이 가장 적합한 모델로 선정되었다.

Claude Sonnet 4.6은 결정사항과 논의사항을 비교적 안정적으로 분리했고, 담당자나 기한이 없는 항목을 임의로 생성하지 않고 “확인 필요”로 표시했다. 또한 중간 조건 변경 이후에도 기존 요구사항을 유지하며 최종 회의록을 재구성했다.

10턴 이상 실행 로그에서는 확정/미확정 구분, 추가 정보 반영, 역할명 사용, 수치 환각 방지, 최종 회의록 작성, 문맥 유지 평가를 수행했다. 최종 평가는 Pass로 판정했다.
